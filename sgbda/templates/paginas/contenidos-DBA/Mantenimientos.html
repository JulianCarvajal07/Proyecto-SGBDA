Motores de Bases de Datos Mantenimientos SQL Server

Guia Mantenimiento

Esta guía presenta un la ejecucion de un mantenimiento para los diferentes clientes con motores de bases de datos sql server. a continuacion se mostra el paso a paso a tener presente:

01

Chequeo de Integridad

El primer paso para realizar un manteniminento sera ejecutar un chequeo de integridad para cada base de datos presente en la instancia debemos tener presente que dicho paso se encuentra automatizado con un archivo ".bat":

Verificar Bases de Datos

ingresar al directorio Documents en el se encontrara un archivo llamado chequeo_integridad.bat o chequeo.bat antes de ejecutar dicho archivo, verificar que coincidan las bases de datos presentes en la instancia con las bases de datos registradas en dicho archivo.

Ejecutar Archivo .bat

luego de la verificacion del archivo chequeo_integridad.bat. debemos dar click derecho en dicho archivo y ejecutar como administrador, se abrira una terminal en el cual empezara a ejecutar dicho archivo, al terminar dicho proceso la ventana se cerrara generando los archivos relacionados con el chequeo teniendo presente la ruta establecida.

02

Reconstruccion y Reorganizacion de Indices

luego de llevar a cabo el chequeo de integridad, se procedera a realizar la Reorganizacion y Reconstruccion de indices para cada instancia presentada en la maquina, debemos tener presente que, para dicho paso, se utilizara otro script para automatizar y optimizar dicho proceso. este script tomara todos los indices creados por el cliente, y dependiento la fragmentacion de cada uno, el script ejecutara una reorganizacion o una Reconstruccion

Reglas

- Se Organizara el indice si su fragmentacion es MENOR a 30.

- Se Reconstruira el indice si su fragmentacion es MAYOR a 30.

- En el caso de presentar error con la Reorganizacion debido a un bloqueo a nivel de página por estar deshabilitado se reconstruira el indice de forma Manual.

Reorganizacion_y_Reconstruccion.sql 

            -- =====================================================================
            -- Script combinado: escaneo de fragmentación + ejecución secuencial por lotes
            -- Ajusta los parámetros al inicio: @reportOnly, @databaseToCheck, @fragmentationThreshold,
            -- @indexFillFactor, @indexStatisticsScanningMode, @sortInTempdb, @verboseMode, @batchSize
            -- =====================================================================

            SET NOCOUNT ON;
            -------------------------------------------------------------
            -- Parámetros de control (ajusta a tu gusto)
            -------------------------------------------------------------
            DECLARE @reportOnly BIT = 0;               -- 1 = solo reporte, 0 = ejecutar
            DECLARE @databaseToCheck SYSNAME = NULL;   -- NULL = todas (o 'TuDB')
            DECLARE @fragmentationThreshold INT = 15;  -- umbral para considerar fragmentado
            DECLARE @indexFillFactor VARCHAR(5) = '90';
            DECLARE @indexStatisticsScanningMode VARCHAR(20) = 'SAMPLED'; -- 'DEFAULT','LIMITED','SAMPLED','DETAILED'
            DECLARE @sortInTempdb VARCHAR(3) = 'ON';
            DECLARE @verboseMode BIT = 0;               -- 1 = prints detallados
            DECLARE @batchSize INT = 1000;             -- "de mil en mil" -> número de comandos por lote
            -------------------------------------------------------------

            -- Validaciones simples
            IF @sortInTempdb NOT IN ('ON','OFF') SET @sortInTempdb='ON';

            -- Preparar tabla temporal de resultados
            IF OBJECT_ID('tempdb..#tmpFragmentedIndexes') IS NOT NULL DROP TABLE #tmpFragmentedIndexes;
            CREATE TABLE #tmpFragmentedIndexes
            (
                [Id] INT IDENTITY(1,1) PRIMARY KEY,
                [dbName] sysname,
                [tableName] sysname,
                [schemaName] sysname,
                [indexName] sysname,
                [databaseID] SMALLINT,
                [objectID] INT,
                [indexID] INT,
                [AvgFragmentationPercentage] FLOAT,
                [reorganizationOrRebuildCommand] NVARCHAR(MAX),
                [Executed] BIT DEFAULT(0),
                [ExecutionResult] NVARCHAR(4000) NULL,
                [ExecutionDate] DATETIME NULL
            );

            -- Variables para validaciones
            DECLARE @isSysAdmin INT;
            DECLARE @SQLServerVersion INT;
            SET @isSysAdmin = (SELECT IS_SRVROLEMEMBER('sysadmin'));
            SET @SQLServerVersion = (SELECT CAST(LEFT(CAST(SERVERPROPERTY('ProductVersion') AS VARCHAR(50)), CHARINDEX('.', CAST(SERVERPROPERTY('ProductVersion') AS VARCHAR(50))) - 1) AS INT));

            IF @isSysAdmin != 1 OR @SQLServerVersion < 9
            BEGIN
                PRINT 'Error: requiere ser sysadmin y SQL Server 2005 o superior.';
                RETURN;
            END

            -- Revisar si AlwaysOn AGs está habilitado
            DECLARE @isHadrEnabled BIT = CAST(ISNULL(SERVERPROPERTY('IsHadrEnabled'),0) AS BIT);

            -- Cursor para bases a escanear (respeta AGs si aplica)
            DECLARE dbNames_cursor CURSOR LOCAL FAST_FORWARD FOR
            SELECT s.[name] AS dbName, s.database_id
            FROM master.sys.databases s
            LEFT JOIN master.sys.dm_hadr_availability_replica_states r ON s.replica_id = r.replica_id
            WHERE s.state_desc = 'ONLINE'
            AND s.is_read_only != 1
            AND s.[name] NOT IN ('master','model','tempdb')
            AND s.[compatibility_level] >= 90
            AND (@databaseToCheck IS NULL OR s.[name] = @databaseToCheck)
            AND (
                    @isHadrEnabled = 0
                    OR (@isHadrEnabled = 1 AND UPPER(ISNULL(r.role_desc,'NonHadrEnabled')) NOT LIKE 'SECONDARY')
                )
            ORDER BY s.database_id;

            OPEN dbNames_cursor;

            DECLARE @dbname sysname;
            DECLARE @dbid INT;
            FETCH NEXT FROM dbNames_cursor INTO @dbname, @dbid;
            WHILE @@FETCH_STATUS = 0
            BEGIN
                IF @verboseMode = 1 PRINT 'Scan: ' + @dbname;

                DECLARE @dynamic_command_get_tables NVARCHAR(MAX) = N'
                USE ' + QUOTENAME(@dbname) + N';
                INSERT INTO #tmpFragmentedIndexes (
                [dbName],[tableName],[schemaName],[indexName],[databaseID],[objectID],[indexID],[AvgFragmentationPercentage],[reorganizationOrRebuildCommand]
                )
                SELECT
                DB_NAME() as [dbName],
                tbl.name as [tableName],
                SCHEMA_NAME(tbl.schema_id) as schemaName,
                idx.name as [indexName],
                pst.database_id as [databaseID],
                pst.object_id as [objectID],
                pst.index_id as [indexID],
                pst.avg_fragmentation_in_percent as [AvgFragmentationPercentage],
                CASE 
                    WHEN pst.avg_fragmentation_in_percent > 30 THEN 
                    ''ALTER INDEX [''+idx.Name+''] ON [''+DB_NAME()+''].[''+SCHEMA_NAME(tbl.schema_id)+''].[''+tbl.name+''] REBUILD WITH (FILLFACTOR = ' + @indexFillFactor + ', SORT_IN_TEMPDB = ' + @sortInTempdb + ', STATISTICS_NORECOMPUTE = OFF);''
                    WHEN pst.avg_fragmentation_in_percent > 5 AND pst.avg_fragmentation_in_percent <= 30 THEN
                    ''ALTER INDEX [''+idx.Name+''] ON [''+DB_NAME()+''].[''+SCHEMA_NAME(tbl.schema_id)+''].[''+tbl.name+''] REORGANIZE;''
                    ELSE
                    NULL
                END
                FROM sys.dm_db_index_physical_stats(DB_ID(), NULL, NULL, NULL, ''' + @indexStatisticsScanningMode + ''') AS pst
                INNER JOIN sys.tables AS tbl ON pst.object_id = tbl.object_id
                INNER JOIN sys.indexes idx ON pst.object_id = idx.object_id AND pst.index_id = idx.index_id
                WHERE pst.index_id <> 0
                AND pst.alloc_unit_type_desc IN (N''IN_ROW_DATA'', N''ROW_OVERFLOW_DATA'')
                AND pst.avg_fragmentation_in_percent >= ' + CAST(@fragmentationThreshold AS VARCHAR(10)) + ';';

                IF @verboseMode = 1
                BEGIN
                    PRINT @dynamic_command_get_tables;
                END

                EXEC (@dynamic_command_get_tables);

                FETCH NEXT FROM dbNames_cursor INTO @dbname, @dbid;
            END

            CLOSE dbNames_cursor;
            DEALLOCATE dbNames_cursor;

            -- Si solo reporte (no ejecutar), mostrar y salir
            IF @reportOnly = 1
            BEGIN
                SELECT dbName, tableName, schemaName, indexName, AvgFragmentationPercentage, reorganizationOrRebuildCommand
                FROM #tmpFragmentedIndexes
                WHERE reorganizationOrRebuildCommand IS NOT NULL
                ORDER BY AvgFragmentationPercentage DESC;
                RETURN;
            END

            -- Si no hay comandos a ejecutar, informar y salir
            IF NOT EXISTS (SELECT 1 FROM #tmpFragmentedIndexes WHERE reorganizationOrRebuildCommand IS NOT NULL)
            BEGIN
                PRINT 'No se encontraron índices que requieran reorganize/rebuild según los parámetros.';
                RETURN;
            END

            -- Ejecutar en lotes de @batchSize. Dentro de cada lote, cada comando se ejecuta secuencialmente.
            DECLARE @totalCommands INT = (SELECT COUNT(*) FROM #tmpFragmentedIndexes WHERE reorganizationOrRebuildCommand IS NOT NULL);
            DECLARE @processed INT = 0;

            PRINT 'Total de comandos a ejecutar: ' + CAST(@totalCommands AS VARCHAR(10));

            WHILE @processed < @totalCommands
            BEGIN
                -- Obtener lote de comandos no ejecutados (ordenados por frag desc)
                ;WITH cteBatch AS
                (
                    SELECT TOP (@batchSize) Id, reorganizationOrRebuildCommand
                    FROM #tmpFragmentedIndexes
                    WHERE reorganizationOrRebuildCommand IS NOT NULL AND Executed = 0
                    ORDER BY AvgFragmentationPercentage DESC, Id
                )
                SELECT * INTO #BatchTemp FROM cteBatch; -- tabla temporal por lote

                DECLARE @cntThisBatch INT = (SELECT COUNT(*) FROM #BatchTemp);
                IF @cntThisBatch = 0
                BEGIN
                    DROP TABLE #BatchTemp;
                    BREAK;
                END

                PRINT '----------------------------------------------------';
                PRINT 'Iniciando lote: procesando ' + CAST(@cntThisBatch AS VARCHAR(10)) + ' comandos. Ya procesados: ' + CAST(@processed AS VARCHAR(10));

                -- Cursor local para ejecutar cada comando dentro del lote (uno a uno)
                DECLARE batch_cursor CURSOR LOCAL FAST_FORWARD FOR
                SELECT Id, reorganizationOrRebuildCommand FROM #BatchTemp ORDER BY Id;

                DECLARE @cmdId INT;
                DECLARE @cmd NVARCHAR(MAX);

                OPEN batch_cursor;
                FETCH NEXT FROM batch_cursor INTO @cmdId, @cmd;
                WHILE @@FETCH_STATUS = 0
                BEGIN
                    PRINT '--- Ejecutando Id=' + CAST(@cmdId AS VARCHAR(10)) + ' ---';
                    PRINT LEFT(@cmd, 200); -- primeros 200 chars para identificar

                    BEGIN TRY
                        EXEC (@cmd);

                        UPDATE #tmpFragmentedIndexes
                        SET Executed = 1,
                            ExecutionResult = 'OK',
                            ExecutionDate = GETDATE()
                        WHERE Id = @cmdId;

                        PRINT '✔ OK Id=' + CAST(@cmdId AS VARCHAR(10)) + ' - ' + CONVERT(VARCHAR(20), GETDATE(), 120);
                    END TRY
                    BEGIN CATCH
                        DECLARE @errMsg NVARCHAR(4000) = ERROR_MESSAGE();
                        PRINT '❌ Error Id=' + CAST(@cmdId AS VARCHAR(10)) + ': ' + @errMsg;

                        -- Marcar como procesado para no bloquear siguiente lote.
                        -- Si prefieres reintentar los fallidos en un futuro, cambia Executed a 0 aquí.
                        UPDATE #tmpFragmentedIndexes
                        SET Executed = 1,
                            ExecutionResult = LEFT('ERROR: ' + @errMsg, 4000),
                            ExecutionDate = GETDATE()
                        WHERE Id = @cmdId;
                    END CATCH;

                    FETCH NEXT FROM batch_cursor INTO @cmdId, @cmd;
                END

                CLOSE batch_cursor;
                DEALLOCATE batch_cursor;

                SET @processed = (SELECT COUNT(*) FROM #tmpFragmentedIndexes WHERE Executed = 1);
                PRINT 'Lote finalizado. Total procesados hasta ahora: ' + CAST(@processed AS VARCHAR(10)) + '/' + CAST(@totalCommands AS VARCHAR(10));

                DROP TABLE #BatchTemp;
            END

            PRINT '----------------------------------------------------';
            PRINT 'Ejecución completa. Resumen:';
            SELECT Id, dbName, schemaName, tableName, indexName, AvgFragmentationPercentage, Executed, ExecutionResult, ExecutionDate
            FROM #tmpFragmentedIndexes
            ORDER BY AvgFragmentationPercentage DESC, Id;

            -- Fin del script
03

Almacenamiento físico y lógico

La organización del almacenamiento determina cómo se distribuyen los datos en disco, cómo se gestiona el crecimiento y cómo se optimiza el rendimiento de I/O. Los tres motores comparten conceptos análogos con nomenclatura distinta.

SQL Server

Filegroups y archivos de datos

Los filegroups agrupan archivos de datos (.mdf primario, .ndf secundarios). Cada base posee un transaction log (.ldf) de escritura secuencial. La base tempdb actúa como área de trabajo global para sorts, tablas temporales y versionado de filas (RCSI).

Oracle

Tablespaces y datafiles

Los tablespaces agrupan datafiles (.dbf). Además de los tablespaces de datos (SYSTEM, SYSAUX, USERS), Oracle requiere tablespaces específicos: UNDO (imágenes previas para rollback y lectura consistente) y TEMP (equivalente a tempdb). Los redo logs rotan en grupos multiplexados.

PostgreSQL

Tablespaces sobre directorios

Los tablespaces de PostgreSQL son directorios del sistema de archivos donde se almacenan los objetos de la base. Los archivos de datos no son visibles directamente por nombre; se identifican internamente por OID. El WAL (Write-Ahead Log) reside en pg_wal/ y cumple la función del transaction log.

Ajuste del mapa mental

Filegroup (SQL Server) = Tablespace (Oracle) = Tablespace (PostgreSQL). El transaction log de SQL Server (.ldf), el redo log de Oracle y el WAL de PostgreSQL son conceptualmente la misma pieza: el registro de cambios previo a la escritura definitiva en el archivo de datos. En Oracle, el modo ARCHIVELOG archiva los redo logs antes de reciclarlos, habilitando la recuperación a un punto específico en el tiempo.

04

Configuración de red y conectividad

La gestión de la capa de red define quién puede conectarse, desde dónde y mediante qué protocolo. Cada motor implementa este control con herramientas y archivos de configuración propios.

SQL Server

Configuration Manager

La herramienta gráfica SQL Server Configuration Manager habilita protocolos (TCP/IP, Named Pipes), fija puertos (1433 por defecto) y gestiona servicios. El SQL Browser resuelve instancias nombradas. La autenticación puede ser de Windows (Kerberos/NTLM) o mixta (SQL Server).

Oracle

Listener y archivos de red

El Listener es un proceso independiente que escucha en el puerto 1521. Se configura en listener.ora. El cliente resuelve conexiones mediante tnsnames.ora. Parámetros generales de red (como DISABLE_OOB para firewalls restrictivos) residen en sqlnet.ora. El diagnóstico de arranque se consulta en listener.log.

PostgreSQL

Archivos de configuración plana

postgresql.conf define la interfaz de escucha (listen_addresses) y el puerto (5432). pg_hba.conf (Host-Based Authentication) controla quién se conecta, desde qué dirección IP y con qué método de autenticación (md5, scram-sha-256, peer, etc.), regla por regla.

05

Backups y recuperación

Las estrategias de respaldo y recuperación comparten objetivos idénticos — proteger la integridad de los datos y habilitar la recuperación a un punto específico — pero difieren en herramientas, nomenclatura y mecánica interna.

SQL Server

BACKUP DATABASE y BACKUP LOG

El modelo de recuperación (Simple / Full / Bulk-logged) determina si es posible realizar backups de log y recuperación punto-a-punto. La cadena de restore sigue el orden: Full → Differential → Log. El historial de backups se almacena en msdb.

Oracle

RMAN y Data Pump

RMAN (Recovery Manager) es una consola cliente dedicada que orquesta backups incrementales (nivel 0 = full, nivel 1 = diferencial) y gestiona el catálogo de backups. El modo ARCHIVELOG es el equivalente al modelo FULL de SQL Server. Data Pump (expdp/impdp) proporciona exportación lógica de schemas o bases completas.

PostgreSQL

pg_dump y pg_basebackup

Los backups se dividen en lógicos (pg_dump, pg_dumpall) y físicos (pg_basebackup). La recuperación punto-en-el-tiempo (PITR) requiere el archivado continuo del WAL. A diferencia de SQL Server y Oracle, PostgreSQL no dispone de una herramienta de backup integrada con catálogo propio; el seguimiento de backups es responsabilidad del administrador o de herramientas externas.

Concepto sin equivalente directo

Flashback de Oracle permite revertir una tabla o la base de datos completa a un estado anterior sin restaurar backups, utilizando la información del tablespace UNDO. SQL Server dispone de Database Snapshots (solo en Enterprise Edition) con mecánica distinta. PostgreSQL no ofrece una funcionalidad nativa análoga al flashback de Oracle.

06

Alta disponibilidad

Las estrategias de alta disponibilidad responden al mismo requerimiento empresarial — minimizar el tiempo de indisponibilidad — pero se implementan con arquitecturas conceptualmente distintas.

SQL Server

Always On y FCI

Always On Availability Groups replica bases de datos de forma síncrona o asíncrona a réplicas secundarias con failover automático. Failover Cluster Instance (FCI) comparte almacenamiento entre nodos, operando en modo activo-pasivo a nivel de instancia.

Oracle

Data Guard y RAC

Data Guard mantiene una base standby (física o lógica) sincronizada mediante redo, con failover manual o automático (Fast-Start Failover). RAC (Real Application Clusters) permite que múltiples instancias accedan simultáneamente a una misma base compartida, operando en modo activo-activo real.

PostgreSQL

Streaming Replication

La replicación por streaming transmite el WAL en tiempo real a un servidor standby, que puede operar en modo hot standby (solo lectura). Para failover automático se requieren herramientas externas como Patroni o repmgr. La replicación lógica permite replicar selectivamente tablas o bases entre servidores.

Diferencia clave

RAC de Oracle no equivale a Always On de SQL Server. RAC es conceptualmente más cercano a un Failover Cluster Instance (varias instancias, una base compartida), mientras que Data Guard es el análogo de Always On Availability Groups (bases separadas, sincronizadas). PostgreSQL no dispone de un equivalente nativo a RAC; su replicación se basa en standby de lectura.

07

Memoria, procesos y concurrencia

La arquitectura de procesos e hilos determina cómo cada motor atiende conexiones concurrentes, gestiona la memoria y responde ante cargas elevadas. Estas diferencias impactan directamente en la configuración del servidor y en el dimensionamiento de recursos.

SQL Server

Multi-hilo con SQLOS

Un único proceso (sqlservr.exe) gestiona múltiples hilos mediante el scheduler de SQLOS. El buffer pool cachea páginas de datos en memoria compartida. El gestor de bloqueos (Lock Manager) controla la concurrencia. El motor administra el crecimiento elástico de hilos sin intervención del administrador.

Oracle

Multi-proceso con SGA compartida

Múltiples procesos background (PMON, SMON, DBWn, LGWR, CKPT) comparten la SGA (System Global Area), una zona de memoria compartida que contiene el buffer cache, el shared pool y el redo log buffer. La concurrencia se gestiona mediante lectura consistente con UNDO, no con bloqueos de lectura.

PostgreSQL

Un proceso por conexión

Cada conexión de cliente genera un backend process independiente del sistema operativo. La memoria compartida se utiliza para el buffer cache y el WAL. La concurrencia se implementa mediante MVCC (Multi-Version Concurrency Control): cada transacción ve una snapshot de la base en el momento de su inicio. Sin un pool de conexiones (PgBouncer), un elevado número de conexiones puede agotar rápidamente los recursos del servidor.

Implicación operativa

En PostgreSQL, el número de conexiones concurrentes debe dimensionarse cuidadosamente, ya que cada conexión consume memoria de proceso independiente. En SQL Server y Oracle, el crecimiento de conexiones es más elástico gracias a la arquitectura multi-hilo y multi-proceso con memoria compartida, respectivamente.
