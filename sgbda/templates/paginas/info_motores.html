01

El choque conceptual más grande: instancia ≠ base de datos

En SQL Server, una instancia es un servidor lógico que hospeda muchas bases de datos independientes. Oracle separa "instancia" (memoria + procesos) de "base de datos" (los archivos físicos) de una forma que no tiene equivalente directo.

Instancia

1 instancia → N bases de datos

Un servicio de Windows (MSSQLSERVER o nombrada) levanta un motor que administra múltiples bases (master, tempdb, y tus bases de usuario) todas visibles desde la misma conexión.

≠
Instance + Database

1 instancia ↔ 1 base (o 1 CDB)

La instancia es el conjunto de memoria (SGA/PGA) y procesos en background (PMON, SMON, DBWn, LGWR...). La database es el conjunto físico de datafiles. Normalmente van pegadas 1 a 1 — salvo en RAC, donde varias instancias comparten una sola base.

Desde Oracle 12c

El modelo default ya no es "una base = un batch de schemas". Es multitenant: una CDB (Container Database, la "instancia física") contiene varias PDB (Pluggable Databases). Cada PDB es lo más parecido a "una base de datos de usuario" en tu mundo SQL Server — es la unidad que conectás, respaldás y clonás de forma independiente.

02

Schema y usuario: en Oracle son la misma cosa

Esta es la trampa más común para un DBA de SQL Server. En tu mundo, Login y Schema son conceptos separados. En Oracle, no.

Login / User / Schema

Tres capas independientes

El LOGIN vive a nivel servidor. Se mapea a un USER dentro de cada base. El SCHEMA (dbo, ventas...) es solo un namespace de objetos — varios usuarios pueden compartir un schema.

≈
User = Schema

Una sola entidad

Cuando hacés CREATE USER juan, automáticamente existe el schema JUAN. No hay forma de separarlos: si el usuario existe, el schema existe, y todo objeto que crea le pertenece a ese schema homónimo.

Nombrado de objetos

Tres partes

[Base].[Schema].[Objeto]
ej: Ventas.dbo.Clientes

≈
Nombrado de objetos

Dos partes

[Schema].[Objeto]
ej: VENTAS.CLIENTES — no hay prefijo de base porque una sesión ya está conectada a una sola (PDB).

Falso amigo

No busques un comando tipo CREATE SCHEMA separado de un usuario en Oracle — existe la sintaxis CREATE SCHEMA AUTHORIZATION, pero solo agrupa varias sentencias DDL en una transacción; no crea un namespace independiente del usuario como en SQL Server.

01.5 — profundización

Multitenant a fondo: CDB y PDB

Esto es lo que más rompe la intuición de un DBA de SQL Server, así que vale detenerse. Desde Oracle 12c (y obligatorio desde 20c en Enterprise), toda instalación nueva usa este modelo por defecto.

Anatomía de una instancia Oracle multitenant1 instancia
CDB — Container Database
la "base física" que la instancia monta. Contiene el diccionario de datos compartido y los metadatos comunes a todo el contenedor.
↓ contiene
CDB$ROOT
metadatos y objetos comunes de Oracle (paquetes PL/SQL del sistema, vistas del diccionario). No guardás datos de aplicación acá.
PDB$SEED
plantilla read-only usada para clonar PDBs nuevas rápidamente — nunca la tocás directamente.
PDB (1..N) — Pluggable Database
acá es donde vive tu "base de datos de negocio". Cada PDB tiene sus propios usuarios, schemas, tablespaces y datafiles — es lo más parecido a una base de usuario de SQL Server.
Tu equivalente mental en SQL Server

Instancia con varias bases

Cuando conectás a un servidor y ves master, tempdb, Ventas, CRM como bases separadas dentro de la misma instancia — esa sensación de "aislamiento lógico, motor compartido" es la que replica el par CDB/PDB.

≈
Diferencia clave

Portabilidad como PK del diseño

Una PDB se puede UNPLUG de una CDB y PLUG en otra — literalmente "enchufar" una base entera en otro servidor físico, algo sin equivalente nativo en SQL Server (lo más cercano: detach/attach de archivos, pero sin el nivel de aislamiento ni portabilidad entre versiones que da Oracle).

Por qué existe esto

El modelo nació para consolidación: en vez de N instancias completas (cada una con su propio SGA, procesos background, overhead de memoria) para N bases pequeñas, corrés una sola instancia grande con N PDBs livianas adentro — parecido en motivación a por qué vos consolidás varias bases chicas en una sola instancia de SQL Server, pero con aislamiento mucho más fuerte entre "bases".

03

Almacenamiento: filegroups vs tablespaces

La jerarquía física es similar en espíritu, pero Oracle agrega un tablespace obligatorio que no tiene equivalente limpio: UNDO.

unidad lógica

↓

archivos físicos de datos

tempdb

↓

TEMP tablespace

Ajustá tu mapa mental

El log de SQL Server es un archivo lineal que truncás con backups de log. En Oracle son grupos de redo logs que rotan (log switch) y, si la base está en modo ARCHIVELOG, cada uno se copia a un archived redo log antes de sobrescribirse — eso es lo que te da recuperación point-in-time, jugando un rol parecido al de tus backups de log pero con una mecánica distinta.

Seguridad y conexión

Autenticación

Windows Auth o SQL Auth

Te conectás a servidor\instancia. El login puede ser de dominio (Kerberos/NTLM) o SQL nativo.

≈
Autenticación

TNS / Service Name

Te conectás a un service name o SID definido en tnsnames.ora, resuelto por el listener de red (puerto 1521 por defecto). El acceso SYSDBA usa un password file separado, no una cuenta "sa".

Cuenta "todopoderosa"

sa

Login con sysadmin a nivel instancia. Casi siempre recomendás deshabilitarlo.

≈
Cuenta "todopoderosa"

SYS / SYSTEM

SYS es dueño del diccionario de datos (equivalente a los catálogos del sistema); SYSTEM es la cuenta administrativa de uso diario. Conectarse como SYS exige AS SYSDBA.

05

Backup & recovery

Herramienta nativa

BACKUP DATABASE / BACKUP LOG

Full, Differential, Log — según el recovery model (Simple / Full / Bulk-logged). Restore encadena Full → Diff → Logs.

≈
Herramienta nativa

RMAN (Recovery Manager)

Motor de backup incremental integrado con catálogo propio. Restore usa RESTORE + RECOVER aplicando archived redo logs, conceptualmente similar a tu cadena Full+Log.

Export lógico

Scripts / BCP / bacpac

Para mover solo datos u objetos, no todo el archivo físico.

≈
Export lógico

Data Pump (expdp / impdp)

Exporta schemas, tablas o la base completa a un formato lógico portable — tu herramienta de referencia cuando no querés mover datafiles enteros.

Concepto sin equivalente directo

Flashback: Oracle permite "retroceder" una tabla o la base entera a un punto anterior sin restaurar backups, usando la información de UNDO. SQL Server no tiene un análogo nativo tan directo (lo más cercano son snapshots de base de datos, que funcionan distinto).

05.5 — profundización

RMAN a fondo

RMAN no es un comando suelto como BACKUP DATABASE: es una herramienta cliente con su propia consola que se conecta a la instancia objetivo (y opcionalmente a un catálogo separado) para orquestar todo el ciclo de backup/recovery.

Cómo se arma un flujo de RMAN
Conexión
rman TARGET / (o TARGET sys/pwd@servicio) — te conectás a la base sobre la que operás, el "target".
↓ opcional
Recovery Catalog
una base separada (o schema) donde RMAN guarda metadatos de todos los backups históricos de varias bases. Sin catálogo, usa el control file de la propia base — suficiente en la mayoría de los casos, igual que vos confiás en msdb para el historial de backups en SQL Server.
Channels
canales de E/S paralelos hacia disco o cinta — es lo que te da paralelismo en el backup, conceptualmente similar a usar múltiples archivos de destino en BACKUP DATABASE ... TO DISK con varios .bak.
Backup incremental nivel 0 / 1
nivel 0 = full baseline. Nivel 1 = solo bloques modificados desde el último backup — el verdadero equivalente a tu Differential, pero acumulable en cadena, no solo contra el último full.
Concepto SQL Server

Recovery Model

Simple / Full / Bulk-logged define si podés hacer backups de log y point-in-time recovery.

≈
Concepto Oracle

ARCHIVELOG mode

Con ARCHIVELOG activado, los redo logs se archivan antes de reciclarse, habilitando recovery a un punto exacto en el tiempo — es el interruptor equivalente a elegir Full en vez de Simple.

Mapa mental rápido

BACKUP INCREMENTAL LEVEL 0 ≈ tu BACKUP DATABASE (Full). BACKUP INCREMENTAL LEVEL 1 ≈ tu BACKUP DATABASE ... WITH DIFFERENTIAL. BACKUP ARCHIVELOG ALL ≈ tu BACKUP LOG. La diferencia mayor: RMAN sabe automáticamente qué bloques cambiaron a nivel de motor (no depende de un bitmap de cambios como el que usa SQL Server para los diferenciales), y todo el catálogo de qué-se-respaldó-cuándo lo mantiene RMAN mismo, no vos a mano.

06

Alta disponibilidad

Clustering activo-activo

— (no existe activo-activo real)

AlwaysOn FCI es activo-pasivo a nivel instancia (almacenamiento compartido).

≈
Clustering activo-activo

RAC (Real Application Clusters)

Varias instancias en distintos nodos acceden a la misma base compartida simultáneamente. Esto es posible justamente porque instancia y base están desacopladas (capa 01).

Réplica / DR

AlwaysOn Availability Groups

Réplicas síncronas/asíncronas a nivel base de datos, con failover automático.

≈
Réplica / DR

Data Guard

Base primaria + standby(s) físicas o lógicas, aplicando redo en modo síncrono o asíncrono. El rol más parecido conceptualmente a un Availability Group.
